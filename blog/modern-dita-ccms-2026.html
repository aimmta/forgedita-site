Modern DITA CCMS

What a modern DITA CCMS should look like in 2026.

A modern DITA CCMS should protect native XML, support Oxygen-first authoring, pin publishing toolchains, expose open APIs, and avoid legacy platform bloat.

Most CCMS platforms were shaped by years of enterprise compromise. Layers were added. Workflows hardened. Admin screens multiplied. Eventually the software started serving itself as much as the people using it.

A modern DITA CCMS should start with a simpler promise: keep DITA as DITA, make the important operations explicit, and do not trap the team inside a platform-shaped maze.

The hard parts are not button colors or dashboards. The hard parts are graph resolution, validation, baselines, publishing reproducibility, clean export, and editor freedom.

Practical take

What this means in the real world.

Native XML is non-negotiable

The repository should preserve DITA/XML as the source of truth, not translate it into a proprietary model and hope nobody notices.

Publishing must be reproducible

A release should pin content, baseline, toolchain bundle, and publishing profile. Anything else is vibes with a PDF button.

Editors should be clients

Oxygen, VS Code, and other tools should talk to open APIs. The editor is not the moat. The operating model is.

Conformance should be public

If a product claims DITA support, it should say exactly what is supported and what is not.
